# 產品需求規格書 (PRD)：醫療貸款送審集中化 Prototype

本產品需求規格書（PRD）專為「醫療專案貸款送審管理系統」的 Prototype（雛形）開發所設計。此系統旨在解決目前業務與 PM 高度依賴分散式 Word 模板、缺乏集中歸檔與進度追蹤的痛點。透過將流程數位化、結構化，並導入直觀的工作看板，加速前端進件與後端審核的推進效率。本文件格式已高度結構化，非常適合直接作為輸入至 Cursor 或 Lovable 等 AI 開發工具的 Prompt，以快速生成前端介面與基礎邏輯。

---

## 1. 專案背景與目標

**痛點：** 目前送審流程高度依賴人工填寫多份 Word 文件，資料散落於各同仁電腦中。不僅版本控制困難、容易發生漏件，跨部門（營業單位與風管徵審）交接時亦缺乏即時的狀態追蹤，導致溝通成本高昂。

**目標：** 建立一個集中的 Web 系統原型，將「客戶資料輸入」、「文件分類歸檔（A、B、C、D 大類）」與「狀態簽核」整合至單一平台，實現無紙化與自動化檢核。

---

## 2. 目標受眾與權限設計 (User Roles)

| 角色 | 職責 |
|------|------|
| **營業單位 (業務/PM)** | 負責新建案件、填寫客戶基本資料、上傳必備文件（執照、財報等），並推進案件至審核階段。 |
| **風管徵審單位** | 負責接收案件、檢閱系統內分類好的徵授信資料與查詢結果，並執行退件、補件或核准動作。 |

---

## 3. 核心功能需求 (Core Features)

### 模組 1：案件建立與動態表單 (Case Management & Dynamic Forms)

**功能描述：** 取代傳統 Word 模板。業務建立新案件時，只需輸入一次客戶核心資料（如企業名稱、負責人、統編），系統便會將這些變數自動帶入所有需要呈現的表單與合約中。

**UI 需求：** 提供「新增案件」按鈕與嚮導式（Wizard）填寫介面，分為「企業基本資料」、「負責人資料」、「貸款條件」三個步驟。

### 模組 2：文件集中歸檔與分類引擎 (Document Hub)

**功能描述：** 實作上一階段討論的分類框架。系統提供專屬的上傳區塊，強制將文件對應至指定的類別。

| 區塊 | 內容 |
|------|------|
| **A 區** | 徵授信資料（授信報核表、申請書等） |
| **B 區** | 查詢資料（聯徵中心、ismart 系統結果截圖或匯出檔） |
| **C 區** | 基本資料（執照、401 表、存摺明細等） |
| **D 區** | 其他資料（身分證影本、同意書等） |

**UI 需求：** 採用拖曳上傳（Drag and Drop）介面，並以手風琴選單（Accordion）或分頁（Tabs）展示 A/B/C/D 四大類。已上傳文件的項目顯示綠色打勾標籤，未上傳顯示紅色警示。

### 模組 3：防呆機制與進度推力設計 (Nudge UI & Validation)

**功能描述：** 在介面設計上導入推力理論（Nudge Theory），引導業務在送件前完成所有必要動作，降低退件率。

**UI 需求：**

- **完成度進度條 (Progress Bar)：** 畫面上方顯示動態進度（例如：資料完整度 85%）。
- **智慧按鈕鎖定：** 若系統檢核發現 A1、C1、D1 等「必備文件」未上傳，則「送交風管」的按鈕將呈現反灰（Disabled）狀態。
- **行動呼籲 (CTA)：** 按鈕旁自動提示「尚缺：負責人身分證影本，上傳後即可送件」。

### 模組 4：看板式狀態追蹤 (Kanban Board)

**功能描述：** 提供一個類似 Trello 的視覺化看板，讓所有人員對案件進度一目了然。

**UI 需求：** 案件以卡片形式呈現，並可於以下直行（Columns）中移動：

1. 資料準備中 (營業單位)
2. 送交檢核中 (風管徵審)
3. 要求補件 (退回營業單位)
4. 審核通過

## 4. 系統資料庫架構雛形 (Data Schema Concept)

Prototype 階段可先採用關聯式資料結構概念：

### Cases (案件表)

`CaseID`, `企業名稱`, `貸款方案`, `當前狀態`, `建立時間`

### Documents (文件表)

`DocID`, `CaseID`, `大類別 (A/B/C/D)`, `文件類型 (如: 401 表)`, `檔案連結`, `上傳時間`

### Users (使用者表)

`UserID`, `姓名`, `部門角色 (業務/風管)`
